# TIL - Java 문법 기초 (2026-06-17)

오늘은 Java의 기본 문법을 처음 배웠다. JavaScript와 비교하며 정적 타입 언어의 특징, 변수와 자료형, 연산자, Scanner를 이용한 입출력까지 다뤘다. (참고: Ex01~Ex06 예제, 강의안 PDF)

## 1. Java는 정적 타입 언어다

- JS는 동적 타입(런타임에 타입 결정)이지만, Java는 변수 선언 시 타입을 명시하고 **컴파일 시점에 타입을 엄격히 검사**한다.
- `javac`가 `.java` 소스를 바이트코드(`.class`)로 컴파일하고, JVM이 이 바이트코드를 실행한다. JS처럼 바로 인터프리트하지 않고 컴파일 단계를 거친다.
- 모든 명령문 끝에 세미콜론(`;`)이 필수다. JS의 세미콜론 자동 삽입 같은 건 없다.
- 모든 코드는 클래스 블록 안에서 시작/종료해야 한다 (전역 함수 없음).

## 2. 진입점 main 메서드

```java
public class HelloWorld {
    public static void main(String[] args) {
        System.out.println("Hello, Java!");
    }
}
```
JVM이 프로그램을 실행할 때 가장 먼저 찾는 규격화된 진입점. `public static void main(String[] args)` 형태를 반드시 지켜야 한다.

## 3. 표기법 (Naming Convention)

| 표기법 | 사용 대상 | 예시 |
|---|---|---|
| camelCase | 변수, 메서드 | `userName`, `calculateTotal` |
| PascalCase | 클래스, 인터페이스 | `HelloWorld`, `OrderService` |
| UPPER_SNAKE_CASE | 상수 | `MAX_LIMIT` |
| kebab-case | Java에서는 변수명에 사용 불가 (URL/HTML/CSS용) | `user-profile` |

## 4. 주석 3종류

- `//` : 한 줄 주석
- `/* ... */` : 여러 줄 주석 (코드 블록 임시 비활성화에도 사용)
- `/** ... */` : Javadoc 문서화 주석. `@param`, `@return` 태그로 메서드 설명을 작성하면 API 문서로 자동 빌드 가능

## 5. 원시 타입(Primitive Type) 8가지

스택 영역에 **값 자체**가 저장된다 (참조 주소 없이 바로 접근, 고속).

| 분류 | 타입 | 크기 | 비고 |
|---|---|---|---|
| 정수형 | byte | 1바이트 | -128 ~ 127 |
| | short | 2바이트 | -32,768 ~ 32,767 |
| | int | 4바이트 | 기본 정수형, 약 ±21억 |
| | long | 8바이트 | 값 끝에 `L` 접미사 필요 |
| 실수형 | float | 4바이트 | `f` 접미사, 7자리 정밀도 |
| | double | 8바이트 | 기본 실수형, 15~16자리 정밀도 |
| 문자형 | char | 2바이트 | 단일 유니코드 문자, `'A'` 형태 |
| 논리형 | boolean | - | `true`/`false`만 가능, 0/1/null과 호환 안 됨 |

- `char`는 내부적으로 정수 코드값을 가져서 `+`, `-` 연산이 가능하다. (`'A' + 1` → 66, `(char)66` → `'B'`)
- int 범위를 넘는 계산이 필요하면 long, long도 넘으면 `BigInteger`.
- 소수점 정밀도 이슈가 중요한 경우엔 `BigDecimal` 사용.

## 6. 참조 타입(Reference Type)과 힙 메모리

- 클래스, 인터페이스, 배열 등 원시 타입이 아닌 모든 것.
- 참조 타입 변수는 스택에 **값이 아니라 힙 영역의 주소값**만 저장한다.
- 모든 클래스는 `java.lang.Object`를 묵시적으로 상속한다.

### String의 불변성(Immutability)
- 문자열은 한 번 생성되면 내용이 바뀌지 않는다.
- `+` 연산을 하면 기존 문자열이 수정되는 게 아니라, **새로운 String 객체가 힙에 생성**되고 참조만 바뀐다.

```java
String msg = "Java";
String updated = msg + " 17"; // 새 객체 생성
System.out.println(msg);      // Java (그대로)
System.out.println(updated);  // Java 17
```

### == 와 .equals() 차이 (오늘 헷갈렸던 부분)
- `==` : 두 변수가 가진 **주소값**을 비교
- `.equals()` : 객체 내부의 **실제 값**을 비교
- 리터럴(`"Java"`)로 만든 문자열은 **String Constant Pool**에서 같은 객체를 공유하므로 `==`가 `true`일 수 있다.
- `new String("Java")`로 만들면 힙에 별도 객체가 생성되므로 `==`는 `false`.
- **결론: 문자열 값 비교는 항상 `.equals()`를 쓴다.**

```java
String a1 = "Java";
String a2 = "Java";
String a3 = new String("Java");
a1 == a2        // true (상수 풀 공유)
a1 == a3        // false (힙에 별도 생성)
a1.equals(a3)   // true (내용은 같음)
```

## 7. 형변환(Casting)과 파싱(Parsing)

- **묵시적 형변환**: 좁은 타입 → 넓은 타입은 자동 변환 (`int → double`)
- **명시적(강제) 형변환**: `(타입)` 으로 강제 변환, 데이터 손실 위험 있음
  - 정수 나누기: `int / int`는 정수 결과만 나오므로, 소수점을 살리려면 피연산자 중 하나를 `double`로 만들어야 함 (`(a * 1.0) / b`)
  - `(int) 3.141592` → `3` (소수점 절삭)
- **문자열 파싱**: `Integer.parseInt("123")`, `Double.parseDouble("3.14")`로 문자열을 숫자로 변환

## 8. Scanner와 입력 버퍼 잔류 문제 (오늘 가장 헷갈렸던 부분)

- `sc.next()` : 공백 기준으로 한 단어만 읽음
- `sc.nextLine()` : 엔터 전까지 한 줄 전체를 읽음
- **버퍼 잔류 버그**: `nextInt()`나 `next()`로 값만 읽으면 입력 버퍼에 개행 문자(`\n`)가 남아있는데, 바로 이어서 `nextLine()`을 호출하면 그 잔류 `\n`을 즉시 읽어버려서 입력을 기다리지 않고 그냥 건너뛴다.
- **해결법**: 숫자/단어 입력 직후에 `sc.nextLine();`을 한 번 단독으로 호출해서 버퍼를 비워준다.

```java
int age = sc.nextInt();
sc.nextLine();              // 잔류 개행문자 제거
String name = sc.nextLine(); // 정상적으로 입력 대기
```

## 9. 연산자

- 산술/대입/비교 연산자는 JS와 거의 동일.
- **문자열 연결(`+`)**: 피연산자 중 하나라도 String이면 나머지를 문자열로 강제 변환. 연산 순서에 주의 (`"" + 10 + 20` → `"1020"`, `"" + (10+20)` → `"30"`).
- **단축 평가**: `&&`, `||`는 앞쪽 결과로 판가름나면 뒤쪽은 평가하지 않음. `&`, `|`는 비트 연산자라 단축 평가가 없음.
- **거듭제곱**: JS의 `**`는 없고 `Math.pow(2, 2)` 사용. `Math.random()`은 0 이상 1 미만.
- **Truthy/Falsy 없음**: `if (1)`, `if (obj)`처럼 boolean이 아닌 값으로 조건을 만들면 컴파일 에러. 반드시 boolean을 반환하는 식을 써야 함.

### 비트 연산 (오늘 가장 헷갈렸던 부분 — 문제·시도·해결·알게된 점)

**문제**
비트 연산자(`<<`, `>>`, `>>>`, `&`, `|`)는 예전부터 나올 때마다 잘 이해가 안 돼서 그냥 넘기곤 했던 부분이다. 오늘 `Ex05.java`를 보면서도 똑같은 지점에서 막혔다. 특히 `>>`와 `>>>`가 둘 다 "오른쪽으로 미는" 연산인데, 왜 `-255 >> 2`는 `-64`가 나오고 `-255 >>> 2`는 `1073741760`처럼 완전히 다른 값이 나오는지 이해가 안 됐다.

**시도**
이번엔 그냥 넘기지 않고 따로 시간을 내서 공부해보기로 했다. Claude한테 비트 연산자를 자세히 설명해달라고 요청했고, Claude가 추천해준 방식대로 직접 8비트 박스를 그려서 비트가 한 칸씩 이동하는 과정을 손으로 따라가보는 연습을 했다.
- `1 << 1`, `1 << 2`를 8개의 비트 칸에 직접 그려보면서, 1이 왼쪽으로 이동할 때마다 오른쪽에 0이 채워지는 걸 눈으로 확인함.
- `-6`처럼 작은 음수를 8비트로 표현해서 `>>`와 `>>>`를 각각 손으로 옮겨보고 두 결과를 나란히 비교함.
- 맨 앞자리(부호비트)가 두 연산에서 어떻게 다르게 채워지는지 따로 표시해가며 확인함.

**해결**
직접 비트를 옮겨보니까 `>>`는 빈자리를 원래 부호비트로 복사해서 채우고, `>>>`는 부호와 무관하게 무조건 0으로 채운다는 차이가 눈에 보였다. 그래서 음수에 `>>>`를 쓰면 부호비트까지 0으로 덮어써지면서 결과가 갑자기 큰 양수로 튀어버리는 것이었다. `-255 >> 2 = -64`(부호 유지) vs `-255 >>> 2 = 1073741760`(부호 무시)도 같은 원리.

**알게된 점**
- 비트 연산은 글로만 읽었을 땐 절대 이해가 안 됐는데, 8비트 박스를 직접 그려서 비트를 한 칸씩 옮겨보는 연습이 확실히 도움이 됐다.
- 머리로만 이해가 안 되는 개념(비트마스킹 등)은 작은 숫자로 직접 손으로 풀어보는 연습을 앞으로도 계속 해야겠다. 예전처럼 그냥 넘기지 않고 이번엔 끝까지 따라가본 게 의미가 있었다.

#### 참고: 왜 그렇게 동작하는지 (원리 정리)

컴퓨터는 모든 정수를 2진수(0과 1, 32비트짜리 `int` 기준)로 저장한다는 게 출발점이다.

#### `<<` 왼쪽 이동 (Shift Left)

비트들을 통째로 왼쪽으로 한 칸씩 밀고, **오른쪽 빈자리는 항상 0으로 채운다**. 한 칸 밀 때마다 값이 2배가 되는 효과가 있다.

```
1 (00000001) << 1  →  2 (00000010)   // 0이 오른쪽에 채워지고 1이 한 자리 위로 이동
2 (00000010) << 1  →  4 (00000100)
```
즉 `n << k` 는 `n * 2^k`와 동일한 결과를 만든다.

#### `>>` 오른쪽 이동 (부호 유지, 산술 이동)

오른쪽으로 밀면서, 빈자리를 **원래 숫자의 부호비트(맨 앞자리)를 그대로 복사**해서 채운다. 그래서 음수는 이동 후에도 계속 음수로 남는다.

#### `>>>` 오른쪽 이동 (부호 무관, 논리 이동)

오른쪽으로 밀면서, 빈자리를 **무조건 0**으로 채운다. 부호비트까지 0으로 덮어쓰기 때문에, 음수에 적용하면 결과가 완전히 다른 (아주 큰) 양수로 바뀐다.

`>>`와 `>>>`의 차이는 음수에서만 드러난다. `-6` (8비트로 표현하면 `11111010`)을 예로 들면:

| 연산 | 빈자리 채움 | 결과 비트 | 결과 값 |
|---|---|---|---|
| `-6 >> 1` (산술) | 부호비트(1) 복사 | `11111101` | `-3` |
| `-6 >>> 1` (논리) | 무조건 0 | `01111101` | `125` |

같은 비트 패턴인데 채우는 방식이 달라서 결과가 음수(`-3`)와 큰 양수(`125`)로 완전히 갈린다. 오늘 예제(`Ex05.java`)에서 `-255 >> 2 = -64`인데 `-255 >>> 2 = 1073741760`로 차이가 컸던 것도 같은 원리다 (단지 `int`는 32비트라 숫자가 훨씬 커질 뿐).

> 핵심 한 줄: **`>>`는 부호를 유지하려고 부호비트를 복사하고, `>>>`는 부호를 무시하고 무조건 0을 채운다.**

#### `&`, `|`, `^`, `~` — 자리수별 비교 연산

| 연산자 | 이름 | 규칙 | 예시 |
|---|---|---|---|
| `&` | AND | 둘 다 1일 때만 1 | `1 & 0 = 0`, `1 & 1 = 1` |
| `\|` | OR | 하나라도 1이면 1 | `1 \| 0 = 1`, `0 \| 0 = 0` |
| `^` | XOR | 서로 다르면 1, 같으면 0 | `1 ^ 0 = 1`, `1 ^ 1 = 0` |
| `~` | NOT | 모든 비트 반전 | `~0 = -1` |

`&`, `|`는 `&&`, `||`와 달리 **단축연산이 없다** — 양쪽 다 항상 평가한다.

#### 비트마스킹 (Bitmasking)

여러 개의 ON/OFF 상태(예: 노드 방문 여부)를 정수 하나의 비트들에 매핑해서 저장하는 기법. `1 << n`으로 "n번째 자리만 1인 마스크"를 만들고, 이걸 조합해서 쓴다.

- **켜기(set)**: `visited = visited | (1 << n)` — OR로 n번째 비트를 1로 만든다.
- **확인(check)**: `(visited & (1 << n)) != 0` — AND로 n번째 비트가 켜져 있는지 본다.

```java
int visited = 0;
visited = visited | (1 << 3);                  // 3번 노드 방문 마킹
boolean isVisited = (visited & (1 << 3)) != 0;  // true
```
배열을 따로 만들지 않고 정수 하나로 여러 상태를 표현할 수 있어서 메모리도 아끼고 연산도 빠르다. 그래서 알고리즘 문제에서 자주 쓰인다고 한다.

## 오늘의 느낀 점 / 헷갈렸던 점

- JS 개발하던 습관으로 `==`를 문자열 비교에 쓰면 안 된다는 걸 다시 체감함. `.equals()` 습관화 필요.
- Scanner의 `nextInt()` → `nextLine()` 조합에서 버퍼 잔류 버그는 직접 실행해보지 않으면 이해하기 어려운 부분이라 실습이 도움이 됐다.
- 정수 나누기에서 소수점이 사라지는 문제 (`a / b`)는 형변환 타이밍을 정확히 알아야 함 — 나누기 *전에* double로 바꿔야 함.
- 비트 연산자는 예전부터 막혔던 부분인데, 오늘은 그냥 넘기지 않고 따로 시간 내서 손으로 그려보며 끝까지 따라가본 게 가장 의미 있었다. (위 "비트 연산" 섹션 참고)
