# 프로그래머스 "다리를 지나는 트럭" 풀이 회고

> 언어: Java | 플랫폼: 프로그래머스 | 자료구조: `Queue<Integer>`

## 문제 한 줄 요약
트럭이 한 줄로 다리를 건넌다. 다리에는 동시에 `bridge_length`대까지만 올라갈 수 있고, 다리 위 트럭들의 무게 합은 `weight`를 넘을 수 없다. 모든 트럭이 다리를 다 건너는 데 걸리는 시간(초)을 구한다.

---

## 1단계. 혼자 풀어본 시도 (약 1시간)

결과: 실패. 코드를 작성하긴 했지만, 머릿속으로 검증하는 과정에서 계속 잘못된 지점이 보여서 끝까지 완성하지 못함.

### 스스로 분석한 실패 원인

1. **문제 이해 부족 (약 30분 손해)**
   다리 위의 트럭이 한 칸씩 이동하면서 1초가 지나고, **마지막 트럭이 다리를 다 건너는 순간까지도 1초가 흘러야 한다**는 점을 늦게 파악함. 문제를 꼼꼼히 읽지 않았고, 문제 상황을 정확히 그려보는 속도가 느렸음.

2. **큐(Queue)를 "안다"와 "쓸 줄 안다"는 다르다는 걸 자각**
   큐라는 자료구조 자체는 알고 있었지만, 이 문제처럼 "다리의 칸"을 표현하는 용도로 실제로 어떻게 활용해야 하는지는 감이 없는 상태였음.

3. **습관적으로 `if-else`로 풀려고 함**
   트럭별로 케이스를 나누는 방식에 익숙해서, "1초당 한 번씩 도는 시뮬레이션 루프"라는 구조보다 익숙한 분기 처리 방식에 먼저 손이 갔음.

## 2단계. AI 힌트를 받아 재도전

> **본인 메모**: 풀이가 길어지자 점점 귀찮아져서 변수명을 너무 대충 지었음 (`isFull`, `exit` 등 의미가 헷갈리는 이름). 변수명 짓는 기본기도 따로 챙겨야 할 부분.

이 단계에서 실제로 작성했던 코드들을 **틀린 버전 → 고친 버전** 순서로 정리한다. (모든 버전은 직접 컴파일·실행해서 결과를 재현했다.)

### 버전 A — 트럭 단위 `for`문 + 즉석 분기 (스크린샷 코드)

```java
for(int truck : truck_weights) {
    sumWeight += truck;
    if (sumWeight < weight && bridge.size() < bridge_length) {
        bridge.offer(truck);
        time++;
    } else if (sumWeight > weight) {
        // 빈 자리 채우고, 한 자리 비우고 트럭 올리기
        ...
    } else if (bridge.size() > bridge_length) {
        ...
    }
}
```

**실행 결과**: Test1 `FAIL (10 / 기대값 8)`, Test2 `PASS (101)`, Test3 `FAIL (109 / 기대값 110)`

**문제점**
- `truck_weights` 배열 기준으로 1바퀴 = "트럭 1대 처리"였는데, 실제 이 문제는 **1초 = 다리 한 칸 전진**이 기본 단위라 구조 자체가 안 맞음.
- "무게는 괜찮지만 다리 자리가 다 찬 경우"를 처리하는 분기가 누락되어, 일부 트럭이 조용히 증발함.

→ **결론**: 트럭 단위 `for`문을 버리고, **1초마다 도는 `while`문**으로 구조를 바꾸기로 함.

### 버전 B — `while`문으로 전환 (1차)

```java
while (!queueTruck.isEmpty() || !bridge.isEmpty()) {
    front = bridge.poll();
    if (front != 0) {
        sumWeight -= front;
        if (!queueTruck.isEmpty()) {
            int next = queueTruck.peek();
            if (sumWeight + next <= weight) {
                next = queueTruck.poll();
                bridge.offer(next);     // ⚠ sumWeight에 next를 더하는 코드가 없음
            } else {
                bridge.offer(0);
            }
        }
        // ⚠ queueTruck이 비었을 때 offer()가 한 번도 안 불림
    } else {
        if (!queueTruck.isEmpty()) {
            int next = queueTruck.peek();
            if (sumWeight + next <= weight) {
                next = queueTruck.poll();
                bridge.offer(next);
            }
            // ⚠ 조건 안 맞을 때 offer(0)이 없음
        } else {
            bridge.offer(0);
        }
    }
    time++;
}
```

**실행 결과**: Test1 → `6` (오답, 기대값 8), Test2·Test3 → **무한루프** (200만 회 강제 중단, `bridge.size()=99`, `sumWeight=-10`에서 고정)

**버그 2가지**
1. **`poll()` 1번에 `offer()` 1번이 짝지어지지 않는 경로가 있음.** 대기 트럭을 다 보낸 뒤 마지막 트럭이 다리에서 빠져나가는 순간 아무것도 `offer()`하지 않아서, 다리 큐 크기가 영구적으로 줄어듦 → 이후로는 빈 칸만 계속 채워서 다리가 끝내 비지 않는 무한루프로 이어짐.
2. **`sumWeight`를 더하는 코드가 빠짐.** 트럭을 올릴 때 `sumWeight += next`가 없어서, 무게 제한 체크(`sumWeight + next <= weight`)가 사실상 무력화됨.

### 버전 C — 위 2가지 버그를 고친 버전

```java
while (!queueTruck.isEmpty() || !bridge.isEmpty()) {
    front = bridge.poll();
    if (front != 0) {
        sumWeight -= front;
        if (!queueTruck.isEmpty()) {
            int next = queueTruck.peek();
            if (sumWeight + next <= weight) {
                next = queueTruck.poll();
                bridge.offer(next);
                sumWeight += next;      // ✅ 추가됨
            } else {
                bridge.offer(0);        // ✅ 추가됨
            }
        } else {
            bridge.offer(0);            // ✅ 추가됨
        }
    } else {
        // (front == 0 분기도 동일하게 offer(0) 보강)
    }
    time++;
}
```

**실행 결과**: 세 테스트 케이스 **모두 무한루프** (200만 회 시점에 `bridge.size()`가 각각 2, 100, 100으로 고정된 채 멈추지 않음)

**새로 드러난 버그**
- `poll()` ↔ `offer()` 짝을 항상 맞췄기 때문에, **다리 큐의 크기가 영원히 `bridge_length`로 고정됨.** 그 결과 `!bridge.isEmpty()`는 `bridge_length > 0`인 한 **절대 `false`가 될 수 없음** → 종료 조건 자체가 성립하지 않는 구조적 문제였음.

→ **결론**: "bridge가 비었는지"가 아니라, **"아직 다리를 다 건너지 못한 트럭이 몇 대인지"**로 종료 조건을 다시 설계해야 함.

### 버전 D — `trucksRemaining` 카운터 추가, 그런데 또 다른 버그

```java
int trucksRemaining = truck_weights.length;

while (!queueTruck.isEmpty() || trucksRemaining > 0) {
    front = bridge.poll();
    if (front != 0) {
        if (!queueTruck.isEmpty()) {
            int next = queueTruck.peek();
            if (sumWeight + next <= weight) {   // ⚠ 아직 front를 안 뺀 sumWeight로 체크
                ...
            }
        }
        sumWeight -= front;   // ⚠ 체크보다 "나중에" 실행됨
        trucksRemaining--;
    }
    ...
}
```

**실행 결과**: Test1 → `10` (오답, 기대값 8), Test2 `101` ✅, Test3 `110` ✅

**버그**: `front`로 빠지는 트럭의 무게를 빼는 시점(`sumWeight -= front`)이 **다음 트럭을 올릴 수 있는지 체크하는 시점보다 뒤에** 와 있었음. 실제 로그로 추적해보면:

```
[time=2] front=7, 빼기 전 sumWeight=7, next=4, sumWeight+next=11 <= 10 ? false
```

7짜리 트럭이 막 빠져나가는 그 순간인데도, 빠진 무게가 아직 반영되지 않아 `7+4=11>10`으로 오판하고 4짜리 트럭을 못 올린 것. 실제로는 `0+4=4<=10`으로 충분히 올릴 수 있어야 했음.

→ **결론**: "트럭이 빠져나간다"는 사실은 "새 트럭을 올릴 수 있는지" 판단하기 **전에** 먼저 반영되어야 함.

## 3단계. 최종 통과 코드 ✅

```java
import java.util.LinkedList;
import java.util.Queue;

public class SolutionTruck {
    public static int solution(int bridge_length, int weight, int[] truck_weights) {
        Queue<Integer> bridge = new LinkedList<>();
        Queue<Integer> queueTruck = new LinkedList<>();

        for (int truck : truck_weights) {
            queueTruck.add(truck);
        }
        for (int i = 0; i < bridge_length; i++) {
            bridge.offer(0);
        }

        int time = 0;
        int sumWeight = 0;
        int front = 0;
        int trucksRemaining = truck_weights.length;

        while (!queueTruck.isEmpty() || trucksRemaining > 0) {
            front = bridge.poll();

            if (front != 0) {
                if (!queueTruck.isEmpty()) {
                    sumWeight -= front;             // 먼저 빼고
                    int next = queueTruck.peek();
                    if (sumWeight + next <= weight) {
                        next = queueTruck.poll();
                        bridge.offer(next);
                        sumWeight += next;
                    } else {
                        bridge.offer(0);
                    }
                } else {
                    bridge.offer(0);
                }
                trucksRemaining--;
            } else {
                if (!queueTruck.isEmpty()) {
                    int next = queueTruck.peek();
                    if (sumWeight + next <= weight) {
                        next = queueTruck.poll();
                        bridge.offer(next);
                        sumWeight += next;
                    } else {
                        bridge.offer(0);
                    }
                } else {
                    bridge.offer(0);
                }
            }
            time++;
        }
        return time;
    }
}
```

**검증 결과**: Test1 `8` ✅ / Test2 `101` ✅ / Test3 `110` ✅ — 추가로 만든 엣지 케이스(`bridge_length=1, weight=10, trucks=[10]` → `2`)와, 독립적으로 작성한 레퍼런스 구현과의 교차검증에서도 일치함을 확인.

## 4단계. 더 정리된 버전 (제안)

로직은 똑같지만, `if(front!=0) / else` 두 블록에서 "다음 트럭 올리기" 코드가 거의 그대로 두 번 반복되고 있어요. 이 둘을 분리하면 다음과 같이 더 짧고 읽기 쉬워져요.

```java
import java.util.LinkedList;
import java.util.Queue;

public class CleanSolution {

    public static int solution(int bridgeLength, int weightLimit, int[] truckWeights) {
        Queue<Integer> bridge = new LinkedList<>();      // 다리 위 칸들 (0 = 빈 칸)
        for (int i = 0; i < bridgeLength; i++) {
            bridge.offer(0);
        }

        int currentWeightOnBridge = 0;
        int nextTruckIndex = 0;
        int totalTrucks = truckWeights.length;
        int elapsedTime = 0;

        // 보낼 트럭이 남아있거나, 다리 위에 트럭이 남아있는 동안 (= 무게가 0보다 큰 동안)
        while (nextTruckIndex < totalTrucks || currentWeightOnBridge > 0) {
            elapsedTime++;

            // 1) 한 칸 전진: 맨 앞 칸이 무조건 빠져나간다
            int exitingTruck = bridge.poll();
            currentWeightOnBridge -= exitingTruck;

            // 2) 다음 트럭을 올릴 수 있으면 올리고, 아니면 빈 칸을 채운다
            boolean hasNextTruck = nextTruckIndex < totalTrucks;
            int nextTruckWeight = hasNextTruck ? truckWeights[nextTruckIndex] : 0;

            if (hasNextTruck && currentWeightOnBridge + nextTruckWeight <= weightLimit) {
                bridge.offer(nextTruckWeight);
                currentWeightOnBridge += nextTruckWeight;
                nextTruckIndex++;
            } else {
                bridge.offer(0);
            }
        }

        return elapsedTime;
    }
}
```

**바뀐 점과 그 이유**

| 항목 | 기존 코드 | 제안 코드 | 이유 |
|---|---|---|---|
| 대기 트럭 표현 | `Queue<Integer>` | `int[]` + 인덱스 | 대기열은 어차피 순서대로만 꺼내 쓰므로, 큐보다 배열 인덱스가 더 가볍고 `peek`/`poll` null 걱정도 원천적으로 없음 |
| 종료 조건 | `trucksRemaining` 카운터 | `currentWeightOnBridge > 0` | 다리 위에 트럭이 남아있다는 사실 자체가 "무게가 0보다 크다"로 이미 표현되어 있어서, 카운터 변수를 따로 안 둬도 됨 |
| 분기 구조 | `if(front!=0)/else`로 "다음 트럭 올리기" 로직이 중복 | "한 칸 전진"과 "다음 트럭 올리기"를 분리 | front가 0이든 트럭이든 "다음 트럭을 올릴지 결정"하는 로직은 동일하므로, 중복 코드를 없애면 버그 날 확률도 줄어듦 |

**검증 결과**: 동일한 3개 테스트케이스 모두 `PASS`.

## 전체적으로 배운 점 (조언)

1. **문제를 읽을 때 "경계 조건"부터 의식하기**: "마지막 트럭이 다리를 다 건너는 순간에도 1초가 흐른다"처럼, 끝나는 지점의 동작을 처음부터 정확히 그려보면 30분을 아낄 수 있었던 부분.
2. **자료구조는 "정의"가 아니라 "역할"로 이해하기**: 큐를 "넣고 빼는 자료구조"로만 알면 활용이 안 됨. 이 문제에서는 "다리의 칸 상태"를 표현하는 역할로 큐를 썼다는 게 핵심.
3. **`poll()`/`offer()` 같은 짝이 되는 연산은 "짝수 법칙"을 의식하기**: 한쪽만 호출되는 경로가 하나라도 있으면 자료구조 크기가 깨지고, 그게 무한루프나 NPE로 이어짐. 디버깅할 때 "이 메서드 호출 뒤에 짝이 되는 호출이 모든 분기에 있는가?"를 체크리스트로 써도 좋음.
4. **연산 순서(빼기 → 체크 vs 체크 → 빼기)가 로직의 의미를 바꿀 수 있음**: "트럭이 빠져나간다"는 사건이 "새 트럭을 올릴 수 있는가"라는 판단보다 먼저 반영돼야 하는 것처럼, 시뮬레이션 문제에서는 이벤트의 순서 자체가 정답/오답을 가른다는 점을 기억.
5. **변수명은 처음부터 의미 있게**: `isFull`, `exit`처럼 상태와 안 맞는 이름을 쓰면 본인도 나중에 헷갈림. `trucksRemaining`, `currentWeightOnBridge`처럼 "무엇을 세고 있는지"가 이름에 그대로 드러나게 짓는 습관이 디버깅 시간을 줄여줌.
6. **막혔을 때는 직접 실행 로그를 찍어서 추적하기**: "왜 안 되지?"를 머릿속 추론만으로 풀기보다, 중간 변수 값을 출력해서 실제 흐름을 보면 가설이 틀렸는지 바로 확인 가능함 (버전 D의 버그도 로그 한 줄로 바로 드러났음).
